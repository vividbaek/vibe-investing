## 1. 작업 ① — 비용 분석 즐겨찾기 + 일일 차트 (5분)

Azure Portal → 검색창에 "비용 관리 + 청구" → "비용 관리" → "비용 분석"

```

### 1.2 첫 화면 권장 설정

다음 4가지를 한 번에 설정하고 즐겨찾기:

| 설정 항목 | 권장값 | 이유 |
|---|---|---|
| 범위 (Scope) | `Azure subscription 1` | 본 프로젝트 구독 단일 |
| 기간 (Time period) | "이번 달까지" 또는 "지난 30일" | 평가판 기간 추적용 |
| 차트 종류 | "누적 영역형" (Stacked area) | 일별 누적 보기 좋음 |
| 분류 (Group by) | "리소스" (Resource) | 8개 리소스 중 어느 게 비용 주범인지 즉시 확인 |

### 1.4 일일 비용 확인 루틴

```
매일 KST 09:00 또는 22:00 (운영자 편한 시간)
1. 즐겨찾기 "증권당 일일 비용" 클릭 (1초)
2. 어제 막대 그래프 색깔 확인:
   - 모두 비슷한 높이 → 정상

## 4. 작업 ④ — 리소스별 태그 적용 (5분)

비용 분석 시 어떤 리소스가 비용 주범인지 명확히 보려면 **태그**를 미리 적용해야 합니다.

### 4.1 진입 경로

각 리소스 → 왼쪽 메뉴 "태그" → 추가

또는 Azure CLI 한 번에:

```bash
# Azure CLI 설치 후 한 번에 8개 리소스에 태그 적용
RG="rg-aiinvestor-prod"
SUB="483f76f5-2ec3-40a0-b8cb-dc3fb67b521e"

# 리소스 그룹 자체에 태그
az group update --name $RG --tags \
  project=jeunggwon \
  environment=prod \
  owner=dennis \
  cost-center=t2e-mvp

# 모든 하위 리소스에 태그 자동 적용
for r in func-aiinvestor-prod plan-aiinvestor-prod swa-aiinvestor-prod \
         appi-aiinvestor-prod kv-aiinvestor-prod law-aiinvestor-prod \
         staiinvestorprod; do
  az resource tag --resource-group $RG --name $r \
    --tags project=jeunggwon environment=prod
done
```

### 4.2 태그 활용

비용 분석에서 "분류" → "태그" → "project" 선택하면 **다른 프로젝트(향후 추가될)와 분리해서 비용 집계** 가능.

### 4.3 권장 태그 4종

| 태그 | 값 | 용도 |
|---|---|---|
| `project` | `jeunggwon` | 프로젝트 단위 비용 집계 |
| `environment` | `prod` | dev/staging/prod 분리 (향후 확장) |
| `owner` | `dennis` | 책임자 명시 (팀 운영 시) |
| `cost-center` | `t2e-mvp` | 회계 분류 (B2B 화이트라벨 시 분리 가능) |

## 7. 작업 ⑦ — 비용 폭주 시나리오 사전 학습 (5분)

평가판 기간에 발생할 수 있는 **비용 폭주 시나리오 5가지**를 미리 알아둡니다.

### 7.1 시나리오 1 — Application Insights 데이터 수집 폭주

**원인**: 모든 함수가 verbose 로그 출력 + 빈번한 호출
**증상**: App Insights 비용이 일 $5+ 발생
**대응**:
```python
# 샘플링 적용 (function_app.py 또는 host.json)
{
  "logging": {
    "applicationInsights": {
      "samplingSettings": {
        "isEnabled": true,
        "maxTelemetryItemsPerSecond": 5
      }
    }
  }
}
```

### 7.2 시나리오 2 — Functions 무한 루프 또는 재귀 호출

**원인**: Timer Trigger 내부에서 자기 자신 호출 / 무한 retry
**증상**: 시간당 수만 회 실행, 일 $10+ 발생
**대응**:
- Functions 최대 인스턴스 수 5로 제한 (앞서 권장)
- Timer 함수에 명시적 종료 조건 추가
- Application Insights에서 "초당 요청 수" 모니터링

### 7.3 시나리오 3 — Storage 트랜잭션 폭주

**원인**: 잘못된 list 작업 (수백만 blob 스캔) 또는 Blob 카운터 오류
**증상**: Storage 트랜잭션 비용이 일 $5+ 발생
**대응**:
- Storage Insights에서 "초당 트랜잭션" 알림 추가
- Lease 기반 카운터의 acquire 실패 무한 retry 방지

### 7.4 시나리오 4 — 외부 API 무한 호출 (LLM 등)

**원인**: 사용자 입력 검증 실패 + 무한 retry / 캐시 미스 루프
**증상**: DeepSeek 비용이 갑자기 $20+/일 발생
**대응**:
- 사용자당 일일 LLM 호출 한도 (200건) 적용
- Functions 응답 시간 30초 timeout 설정
- 외부 API 실패 시 exponential backoff (1, 2, 4, 8초)

### 7.5 시나리오 5 — Key Vault 호출 폭주

**원인**: 모든 함수 호출마다 Key Vault에서 secret 가져오기 (캐싱 안 함)
**증상**: KV 작업 한도 (월 10,000) 초과 + 비용 발생
**대응**:
- Functions 인스턴스 시작 시 1회만 KV 호출, 이후 메모리 캐시
- 또는 환경 변수로 secret 주입 (일부 secret만)

### 7.6 비용 폭주 발생 시 긴급 조치 절차

```
1. Cost Management 알림 수신 (이메일)
2. 비용 분석 화면 진입 → "리소스" 분류 → 어느 리소스 폭주인지 확인
3. 해당 리소스 일시 정지:
   - Functions: 함수 앱 → "중지" 버튼 (즉시 모든 실행 중단)
   - Static Web Apps: 영향 거의 없음, 그대로 두기
   - Storage: 절대 중지 안 함 (데이터 유지)
4. 원인 분석 → 코드 수정 → 재배포 → 재시작
```

| **B. 매우 낮음** | $0~5 | 평가판 크레딧 거의 안 씀, 즉시 종량제 전환 + 안심 |
| **C. 매우 높음 (예상치 못함)** | $50+ | 비용 분석 + 비정상 리소스 식별 후 코드 수정 → 종량제 전환 |
| **D. 한국 거주자 정책 변경 등** | 외부 변수 | 전환 보류, 다른 클라우드 검토 |

대부분의 베타 시나리오는 **A 또는 B**입니다.

### 8.2 종량제 전환 절차

```
1. Azure Portal → "구독" → "Azure subscription 1"
2. "구독 변경" 또는 "종량제 요금제로 업그레이드"
3. 결제 정보 추가 (신용카드)
4. 전환 즉시:
   - 12개월 무료 서비스 그대로 유지
   - 평가판 크레딧 잔액 (있다면) 그대로 사용 가능
   - 무료 한도 초과분만 종량제 청구
```

### 8.3 종량제 전환 후 첫 1주일 모니터링 강화

| 점검 항목 | 빈도 |
|---|---|
| 일일 비용 확인 | 매일 (아침 1회) |
| 무료 한도 사용률 | 매일 (저녁 1회) |
| 알림 수신 확인 | 즉시 (이메일 즉시 처리) |
| 비정상 트래픽 패턴 | 매일 |

전환 후 첫 1주일이 안정되면 이후는 주 1회 점검으로 전환.

## 9. 솔직한 한계 메모

- **비용 데이터는 24~48시간 지연**: Cost Management의 일일 비용은 어제 데이터까지만 확정. 오늘 비용은 다음날 확인 가능.
- **평가판 크레딧과 무료 한도는 별개**: 평가판 사용 중에는 모든 비용이 크레딧에서 차감되어 무료 한도 적용 여부가 잘 안 보임. 종량제 전환 후에야 무료 한도가 명확히 작동.
- **알림 발송에 약간의 지연**: 임계값 도달 후 5~30분 지연 후 이메일 발송. 진짜 폭주 시에는 이미 추가 비용 발생.
- **태그 적용 후 즉시 비용 집계 안 됨**: 태그 적용 시점부터 미래 비용만 태그별 분류. 과거 비용은 분류 안 됨.
- **Telegram Webhook 알림은 추가 설정 필요**: 액션 그룹의 webhook은 페이로드 형식이 표준화되어 있어 Telegram이 그대로 받지 못할 수 있음. 중간에 Logic App 또는 Function 추가 필요.
